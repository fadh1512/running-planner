## Project Overview

A full-stack web application that combines **running training** and **strength training** into one intelligent planner. Users can schedule workouts, track progress, log recovery, and get training statistics.

### Features Built

| Feature | Status | Description |
|---------|--------|-------------|
| Dashboard | Done | Today's workout, weekly stats, streak, PRs |
| Calendar | Done | Monthly view with color-coded workouts, create/edit/complete |
| Workout CRUD | Done | Create, update, delete, complete workouts |
| Running Templates | Done | 7 pre-built running workouts (Easy, Tempo, Intervals, etc.) |
| Strength Templates | Done | 3 pre-built strength routines (Push, Pull, Legs & Core) |
| Progress Tracking | Done | Weekly distance charts, workout history, PR tracking |
| Statistics | Done | Running stats, strength stats, streaks |
| Recovery Tracker | Done | Daily log with calculated recovery score (0-100%) |

## Setup Guide

### Prerequisites
- Linux Mint / Ubuntu (the setup script is for apt-based systems)
- Internet connection

### Quick Setup (Recommended)

```bash
cd "/home/fadh/Documents/Running Planner"
chmod +x setup.sh
./setup.sh
```

This installs everything: Node.js, PostgreSQL, Python packages, npm packages, and creates start scripts.

### Manual Setup

If you prefer to install things yourself:

#### 1. Install Node.js
```bash
curl -fsSL https://deb.nodesource.com/setup_18.x | sudo -E bash -
sudo apt-get install -y nodejs
```

#### 2. Install PostgreSQL
```bash
sudo apt install postgresql postgresql-contrib libpq-dev
sudo systemctl start postgresql
```

#### 3. Install Python venv
```bash
sudo apt install python3.12-venv python3-pip
```

#### 4. Create Database
```bash
sudo -u postgres psql -c "CREATE DATABASE running_planner;"
```

#### 5. Setup Backend
```bash
cd backend
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt
cp .env.example .env
# Edit .env if your postgres password is not "password"
```

#### 6. Setup Frontend
```bash
cd frontend
npm install
```

### Starting the App

**Option A - Both at once:**
```bash
./start-all.sh
```

**Option B - Separate terminals:**
```bash
# Terminal 1 - Backend
./start-backend.sh

# Terminal 2 - Frontend
./start-frontend.sh
```

Then open **http://localhost:5173** in your browser.

| URL | What |
|-----|------|
| http://localhost:5173 | Frontend app |

## Database Schema

### Tables

#### 1. `workouts`
The main workout table. Every scheduled workout is stored here.

| Column | Type | Description |
|--------|------|-------------|
| id | INTEGER | Primary key |
| title | VARCHAR(255) | Workout name |
| workout_type | ENUM | Type: easy_run, tempo_run, interval, hill_repeats, long_run, progression_run, recovery_run, strength_a, strength_b, strength_c, rest_day, custom |
| date | DATE | Scheduled date |
| completed | BOOLEAN | Whether the workout was done |
| completed_at | DATETIME | When it was marked complete |
| notes | TEXT | User notes |
| estimated_duration | INTEGER | Estimated minutes |
| created_at | DATETIME | Record creation time |
| updated_at | DATETIME | Last update time |

#### 2. `run_details`
Extended details for running workouts (1:1 with workouts).

| Column | Type | Description |
|--------|------|-------------|
| id | INTEGER | Primary key |
| workout_id | INTEGER | FK to workouts (unique) |
| distance | FLOAT | Distance in km |
| target_pace | VARCHAR(20) | Target pace (e.g., "5:30") |
| actual_pace | VARCHAR(20) | Actual pace after completion |
| duration | INTEGER | Duration in minutes |
| elevation_gain | FLOAT | Elevation in meters |
| heart_rate_avg | INTEGER | Average heart rate |

#### 3. `strength_details`
Exercise entries for strength workouts (1:many with workouts).

| Column | Type | Description |
|--------|------|-------------|
| id | INTEGER | Primary key |
| workout_id | INTEGER | FK to workouts |
| exercise_name | VARCHAR(255) | Exercise name |
| sets | INTEGER | Number of sets |
| reps | INTEGER | Reps per set |
| weight | FLOAT | Weight in kg |
| rest_time | INTEGER | Rest time in seconds |
| notes | TEXT | Exercise notes |

#### 4. `workout_templates`
Pre-built running workout templates.

| Column | Type | Description |
|--------|------|-------------|
| id | INTEGER | Primary key |
| name | VARCHAR(255) | Template name |
| category | ENUM | Workout type |
| description | TEXT | Description |
| warmup | TEXT | Warm-up instructions |
| main_workout | TEXT | Main set instructions |
| cooldown | TEXT | Cool-down instructions |
| target_pace | VARCHAR(20) | Target pace |
| estimated_duration | INTEGER | Estimated minutes |

#### 5. `strength_templates`
Pre-built strength workout templates.

| Column | Type | Description |
|--------|------|-------------|
| id | INTEGER | Primary key |
| name | VARCHAR(255) | Template name |
| template_type | VARCHAR(50) | A, B, or C |
| description | TEXT | Description |

#### 6. `strength_template_exercises`
Exercises within strength templates (1:many with strength_templates).

| Column | Type | Description |
|--------|------|-------------|
| id | INTEGER | Primary key |
| template_id | INTEGER | FK to strength_templates |
| exercise_name | VARCHAR(255) | Exercise name |
| default_sets | INTEGER | Default sets |
| default_reps | INTEGER | Default reps |
| default_weight | FLOAT | Default weight in kg |
| notes | TEXT | Notes |

#### 7. `training_plans`
User training plans (e.g., 12-week marathon plan).

| Column | Type | Description |
|--------|------|-------------|
| id | INTEGER | Primary key |
| goal | VARCHAR(50) | 5k, 10k, half_marathon, marathon |
| start_date | DATE | Plan start |
| end_date | DATE | Race day |
| current_weekly_km | FLOAT | Starting weekly distance |
| fitness_level | VARCHAR(50) | beginner, intermediate, advanced |
| training_days | INTEGER | Days per week |
| strength_sessions | INTEGER | Strength sessions per week |
| active | BOOLEAN | Is this the active plan? |

#### 8. `personal_records`
Track best performances.

| Column | Type | Description |
|--------|------|-------------|
| id | INTEGER | Primary key |
| category | VARCHAR(100) | e.g., fastest_5k, best_squat, best_bench |
| value | FLOAT | The record value |
| unit | VARCHAR(20) | min, kg, km |
| achieved_at | DATE | Date achieved |

#### 9. `recovery_logs`
Daily recovery check-ins.

| Column | Type | Description |
|--------|------|-------------|
| id | INTEGER | Primary key |
| date | DATE | Log date (unique) |
| sleep_hours | FLOAT | Hours of sleep |
| energy_level | INTEGER | 1-10 scale |
| muscle_soreness | INTEGER | 1-10 scale |
| stress_level | INTEGER | 1-10 scale |
| recovery_score | FLOAT | Calculated 0-100 |
| notes | TEXT | Notes |

### Relationships
```
workouts 1──1 run_details
workouts 1──N strength_details
strength_templates 1──N strength_template_exercises
```

## Backend API Reference

### Base URL: `http://localhost:8000`

### Workouts `/api/workouts/`

| Method | Endpoint | Body | Description |
|--------|----------|------|-------------|
| GET | `/api/workouts/` | — | List all workouts (optional ?start_date=&end_date=) |
| GET | `/api/workouts/{id}` | — | Get one workout |
| POST | `/api/workouts/` | WorkoutCreate | Create a new workout |
| PUT | `/api/workouts/{id}` | WorkoutUpdate | Update a workout |
| PATCH | `/api/workouts/{id}/complete` | — | Mark as completed |
| DELETE | `/api/workouts/{id}` | — | Delete a workout |

**Create Workout Example:**
```json
{
  "title": "Morning Easy Run",
  "workout_type": "easy_run",
  "date": "2026-07-19",
  "notes": "Felt great",
  "estimated_duration": 40,
  "run_details": {
    "distance": 5.0,
    "target_pace": "5:30"
  }
}
```

**Create Strength Workout Example:**
```json
{
  "title": "Strength A",
  "workout_type": "strength_a",

    {"exercise_name": "Squat", "sets": 4, "reps": 8, "weight": 60},
```

### Recovery Score Algorithm

The recovery score is calculated as follows:
```
Base score = 50

+ Sleep:  8+ hours = +20, 6-8 = +10, <5 = -10
+ Energy: (value - 5) * 3
+ Soreness: (5 - value) * 3  (less sore = better)
+ Stress: (5 - value) * 2   (less stress = better)

Result clamped to 0-100
```

Score interpretation:
- **70-100%**: Ready for hard training
- **40-69%**: Consider easy training or rest
- **0-39%**: Rest day recommended

## How It Works

### Data Flow
```
User clicks in UI
    ↓
React component calls api/client.ts function
    ↓
HTTP request to FastAPI backend (/api/*)
    ↓
Router handler in routers/*.py receives request
    ↓
crud.py performs database operations via SQLAlchemy
    ↓
PostgreSQL stores/returns data
    ↓
Response flows back to frontend
    ↓
React state updates, UI re-renders
```

### Auto-Seeding
On first startup, the backend automatically:
1. Creates all database tables
2. Seeds 7 running workout templates (Easy, Tempo, Intervals, Hills, Long, Progression, Recovery)
3. Seeds 3 strength templates (A: Push, B: Pull, C: Legs & Core)

### Vite Proxy
The frontend dev server (port 5173) proxies all `/api/*` requests to the backend (port 8000). This avoids CORS issues during development.

## Phase 3 Features

### Training Plan Generator (Fixed)

The plan generator now creates actual workouts in the calendar when you generate a plan.

**How it works:**
1. User selects goal (5K/10K/HM/Marathon), dates, fitness level, training days
2. Backend calculates a weekly schedule with proper workout types
3. Generates progressive workouts (distance increases over time)
4. Includes a 2-week taper period before race day
5. All workouts appear in the calendar immediately

**Weekly pattern** (5 training days example):
- Mon: Easy Run
- Tue: Interval Session
- Wed: Strength A
- Thu: Tempo Run
- Sat: Long Run

**Progression logic:**
- Weeks 1-N: Distance increases from 70% to 100% of target
- Taper (last 2 weeks): Distance drops to 60% of target
- Long run distance scales with goal (e.g., marathon: 24-35km)

**File:** `backend/app/plan_generator.py`

### Personal Records Auto-Detection

PRs are automatically detected when a workout is completed:

| Trigger | PR Category |
|---------|-------------|
| Run distance > previous longest | `longest_run` |
| Run pace < previous fastest | `fastest_pace` |
| Weekly mileage > previous best | `highest_weekly_km` |
| Squat weight > previous best | `best_squat` |
| Bench press weight > previous best | `best_bench` |
| Deadlift weight > previous best | `best_deadlift` |
| Pull-up weight > previous best | `best_pullup` |

**File:** `backend/app/crud.py` → `_check_running_prs()`, `_check_strength_prs()`

### Mobile Responsive Design

The frontend is fully responsive:
- **Sidebar**: Hamburger menu on mobile, slide-in overlay
- **Dashboard**: 2-column stats on mobile, 4-column on desktop
- **Calendar**: Touch-friendly, bottom-sheet modals on mobile
- **All pages**: Responsive grids, stacked layouts on small screens
- **Modals**: Bottom sheet on mobile, centered on desktop

### UI/UX Improvements

- **Gradient buttons** with hover animations
- **Card hover effects** with subtle shadow transitions
- **Loading spinners** instead of text
- **Color-coded stat boxes** (blue for distance, orange for strength, etc.)
- **Progress bars** with gradients
- **Score circle** for recovery (green/yellow/red)

## Deployment Guide

### Frontend (Vercel)

1. Push your code to GitHub
2. Go to [vercel.com](https://vercel.com) and sign in with GitHub
3. Click "New Project"
4. Import your repository
5. Set the root directory to `frontend`
6. Vercel auto-detects Vite - click "Deploy"
7. After deployment, go to Settings → Environment Variables
8. No env vars needed for frontend (it uses relative `/api` paths)

**Important:** After deploying frontend, update the Vite proxy to point to your backend URL, OR configure the API client to use the full backend URL.

### Backend (Railway)

1. Go to [railway.app](https://railway.app) and sign in with GitHub
2. Click "New Project" → "Deploy from GitHub repo"
3. Select your repository
4. Set the root directory to `backend`
5. Railway auto-detects Python and the Dockerfile
6. Add environment variable: `DATABASE_URL` = your PostgreSQL connection string
7. Deploy

**For PostgreSQL on Railway:**
1. In the same Railway project, click "New" → "Database" → "PostgreSQL"
2. Railway auto-generates a `DATABASE_URL` variable
3. The backend will use it automatically

### Backend (Render)

1. Go to [render.com](https://render.com) and sign in
2. Click "New" → "Web Service"
3. Connect your GitHub repo
4. Settings:
   - Root Directory: `backend`
   - Build Command: `pip install -r requirements.txt`
   - Start Command: `uvicorn app.main:app --host 0.0.0.0 --port $PORT`
5. Add PostgreSQL database in Render dashboard
6. Set `DATABASE_URL` environment variable

### Backend (Fly.io)

1. Install flyctl: `curl -L https://fly.io/install.sh | sh`
2. In the `backend/` directory: `fly launch`
3. `fly deploy`
4. Add a Postgres database: `fly postgres create`
5. Connect it: `fly secrets set DATABASE_URL=...`

### Environment Variables for Production

| Variable | Value | Where |
|----------|-------|-------|
| `DATABASE_URL` | `postgresql://user:pass@host:5432/db` | Backend |
| `CORS_ORIGINS` | Your frontend URL | Backend (optional) |

### Updating Frontend API URL for Production

If your frontend and backend are on different domains, update `frontend/src/api/client.ts`:

```typescript
const BASE = import.meta.env.VITE_API_URL || '/api';
```

Then in Vercel, set `VITE_API_URL` to your backend URL (e.g., `https://your-app.up.railway.app/api`).

## Troubleshooting

### "Database connection refused"
- Make sure PostgreSQL is running: `sudo systemctl start postgresql`
- Check your password in `backend/.env`

### "ModuleNotFoundError" on backend
- Make sure you activated the virtual environment: `source backend/venv/bin/activate`
- Reinstall: `pip install -r requirements.txt`

### "npm: command not found"
- Reinstall Node.js: `curl -fsSL https://deb.nodesource.com/setup_18.x | sudo -E bash - && sudo apt install -y nodejs`

### Frontend shows "Failed to fetch"
- Make sure backend is running on port 8000
- Check Vite proxy in `frontend/vite.config.ts`
